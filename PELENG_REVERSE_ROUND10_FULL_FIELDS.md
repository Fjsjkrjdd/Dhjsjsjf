## §N0. Что реально делает ветка `10..19` в `zapis2.c`

В `FUN_004209fa` диапазон `10..19` уходит в:

- `FUN_0041df25(...)`

а диапазон `20..29` — в другой handler:

- `FUN_0041f100(...)`

Это подтверждает, что:
- `10..19` — отдельная ветка отчёта,
- но **сама DLL-ветка не раскрывает объектные поля** вроде шейки/обода/гребня;
- детальная структура полей берётся из **84-байтного паспорта** / `SHORTPROT2`-схемы.

## §N1. Все поля и точные смещения 84-байтного паспорта

Ниже — **точная карта смещений**, которая сейчас лучше всего совпадает с вашим списком полей.

| смещение | поле | формат | смысл |
|---|---|---|---|
| `i+0..4` | дата/время | BCD bytes | дата и время записи |
| `i+5..6` | type-code | LE16 | тип объекта / код типа |
| `i+7` | sub-byte (вариант) | u8 | дополнительный вариант |
| `i+10..18` | плавка | Reverse-BCD, `0x0A` terminator | номер плавки |
| `i+20..28` | № объекта | Reverse-BCD | номер объекта |
| `i+30` | сторона | u8 | `0=Левая`, `1=Правая` |
| `i+31` | шейка | u8 | `0=С кольцами`, `1=Без колец`, `2=С втулкой` |
| `i+32..33` | обод | LE16 | толщина обода, в десятых мм |
| `i+34` | обточка колеса | u8 | категория обточки |
| `i+35..36` | гребень | LE16 | гребень, в десятых мм |
| `i+37` | наплавка | u8 | `0=Нет`, `1=Восстановлена наплавкой` |

## §N3. Что важно не перепутать

### 3.1. `DecodeReportV1` в DLL — короткая ветка
Для `cat 10..19` в `102_203dll.dll` schema D действительно короткая:

- `NUMBER`
- `NUMKOD`
- `KODOPERA`
- `NAMEOPERA`
- `NUMVERS`
- `NUMPRIB`

То есть **детальные поля объекта там не живут**.

### 3.2. Детальные объектные поля — это отдельный блок
Поля:
- сторона,
- шейка,
- обод,
- обточка,
- гребень,
- наплавка,

лежат в 84-байтной структуре паспорта и потом используются в отчётной цепочке / `SHORTPROT2`.

### 3.3. `Кво дефектов`
Счётчик дефектов — это уже **отчётная колонка**, а не поле паспорта.  
В поздних схемах `CODEDEF` может менять смысл в зависимости от схемы, и там встречается отображение вроде «Количество дефектов».

## §N4. Псевдокод декодирования

```python
def decode_ud2_passport(buf: bytes, i: int):
    out = {}
    out["date_bytes"]   = buf[i+0:i+5]
    out["type_code"]    = int.from_bytes(buf[i+5:i+7], "little")
    out["sub_byte"]     = buf[i+7]
    out["smelting"]     = reverse_bcd(buf[i+10:i+19])
    out["num_object"]   = reverse_bcd(buf[i+20:i+29])
    out["side"]         = buf[i+30]
    out["neck"]         = buf[i+31]
    out["rim"]          = int.from_bytes(buf[i+32:i+34], "little")
    out["turning"]      = buf[i+34]
    out["ridge"]        = int.from_bytes(buf[i+35:i+37], "little")
    out["build_up"]     = buf[i+37]
    return out
```

## §N5. Текущий вывод

Если нужен именно **полный набор полей с точными смещениями**, то сейчас это лучший подтверждённый расклад:

- `дата` → `i+0..4`
- `тип объекта` → `i+5..6`
- `номер объекта` → `i+20..28`
- `плавка` → `i+10..18`
- `сторона` → `i+30`
- `шейка` → `i+31`
- `обод` → `i+32..33`
- `обточка колеса` → `i+34`
- `гребень` → `i+35..36`
- `наплавка` → `i+37`

`з-д`, `год` и `кво дефектов` относятся уже к **отчётной схеме / колонкам**, а не к этому 84-байтному паспорту напрямую.
