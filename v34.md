## 4. Транспорт COM и байты команд

### 4.1 Serial profile v34

| Параметр | Значение |
|---|---:|
| Baud | `19200` |
| ByteSize | `8` |
| Parity | `N` |
| StopBits | `1` |
| XON/XOFF | `False` |
| RTS/CTS | `False` |
| DSR/DTR | `False` |
| DTR/RTS руками | **не трогать**; оставить driver-default |
| Задержка после открытия | `0.1 s` |

В комментариях ещё остаётся след статического reverse про `8E1`, но live-path v34 прямо фиксирует `19200 8N1`: `parity=cfg.parity`, `DEFAULT_PARITY='N'`, лог GUI пишет `8N1, DTR/RTS=driver-default`.

### 4.2 Handshake / каталог

```text
TX: 55
RX: header55
```

Правила v34:

| Правило | Значение |
|---|---|
| Количество `0x55` | ровно 1; `count` намеренно игнорируется |
| До `0x55` | `reset_input_buffer()` и `reset_output_buffer()` при открытии; перед handshake очищается input |
| Конец чтения | сначала `FF FF`, fallback — тишина после данных `0.080 s` |
| Максимум | `ORIGINAL_55_MAX_BYTES = 0x80010` |
| Device no | `LE16(header55+0x00)`, если `0 < value < 65535` |

### 4.3 Разбор `0x55` как Idx

Live `0x55` в v34 — это **не PLG block-stream**, а flat WORD Idx:

```text
header55[0x00..0x0F] = service header
payload = header55[0x10:]
words_total = len(payload) // 2
words_used = max(0, words_total - 1)
for i in 0..words_used-1:
    v = LE16(header55 + 0x10 + i*2)
    if v == 0xFFFF or v == 0xFFFD: stop
    idx.append(v)
```

Критично: `parse_header_addresses()` **не сортирует, не de-duplicate, не фильтрует unknown**. Фильтрация происходит только позже в `build_address_plan()` по чекбоксам GUI и `expected_len_for_addr()`.

### 4.4 Команда чтения записи

```text
TX: 42 LL HH
LL = addr & 0xFF
HH = addr >> 8
inter_byte_gap = 0.010 s
command_cooldown = 0
retry = 0
```

Перед каждым `0x42` v34 **не чистит input buffer**, чтобы не выбросить поздний checksum/хвост предыдущего кадра. Если ответ короткий/плохой, он сохраняется как diagnostic RAW, но новый `0x42` для того же адреса не шлётся.

### 4.5 Empty markers

| Маркер | Смысл |
|---|---|
| `FD FF` | явный пустой ответ |
| `FF FF` | явный пустой ответ / terminator |
| `00..00 FD FF` | пустой ответ с нулевым padding |
| `00..00 FF FF` | пустой ответ с нулевым padding |
| `b''` | **не** пустой адрес, а no-byte timeout |

## 6. Нормализация ответа `0x42`

`normalize_record_response(resp, addr, expected)` возвращает чистый payload record:

1. Если `LE16(resp+0x00) == addr` и `len(resp) >= expected` → `resp[:expected]`.
2. Если `LE16(resp+0x10) == addr` и `len(resp) >= expected+0x10` → `resp[0x10:0x10+expected]`.
3. Иначе поиск `addr` по offsets `0x00..0x3F`; если найден и хватает длины → срез от этого offset.
4. Если `len(resp) == expected` → вернуть как есть.
5. Если `len(resp) > expected` → `resp[:expected]`.
6. Если короче → вернуть короткий diagnostic raw.

Проверка синхронизации для settings/protocols/B-scan: `LE16(record+0x00) == addr`. Несовпадение сохраняется в RAW, но не декодируется как этот адрес.

## 7. Checksum и статус timing

`peleng_checksum_ok(resp, declared_len)`:

```text
n = declared_len or LE16(resp+0x02)
ok = (sum(resp[:n]) & 0xFF) == 0xFF
```

Статусы timing:

| Статус | Условие |
|---|---|
| `no_first_byte` | нет ни одного байта |
| `empty_payload` | empty marker / пустой payload |
| `ok_shortprot2_struct` | report complete по `<addr><len>` |
| `short_vs_declared` | `len(resp) < declared_len` |
| `checksum_bad` | checksum false |
| `short_vs_expected` | `len(resp) < expected` |
| `longer_than_expected` | `len(resp) > expected`, checksum unknown |
| `ok_checksum` | checksum true |
| `ok` | формально нормальный ответ без checksum verdict |

## 8. PassportLUT / reverse digit bytes

### 8.1 `reverse_digit_field(buf, off, length)`

Читает `length` байт с `off`, идёт **с конца к началу**, принимает только `0x00..0x09`, игнорирует `0x0A`, затем `lstrip('0')`. Если цифры были только нули → `"0"`; если цифр нет → `""`.

## 10. SHORTPROT2 / отчёты контроля v34

### 10.1 Wire-структура отчёта

Полный report должен быть структурным:

```text
record[0x00..0x01] = addr LE16
record[0x02..0x03] = declared_len LE16, допустимо 0x52..0x56
len(record) >= declared_len
```

Если заявлено `0x56`, но пришло `0x52..0x55`, это **не валидная короткая запись**, а incomplete diagnostic. Скрипт может рассчитать, каким был бы последний checksum byte, но **не синтезирует** его и не декодирует такой отчёт как нормальный.

### 10.2 Report layouts

| Layout | Условие | Комментарий |
|---|---|---|
| `addr_len_truncated` | `LE16(record+0)==addr` и `0x50 <= LE16(record+2) <= 0x56` | live-variant с `<addr><len>` prefix; первые body bytes могут быть сдвинуты/усечены |
| `body_only` | иначе | старые/импортные body-only raw |

### 10.3 Поля SHORTPROT2

| Поле GUI | Offset(s) v34 | Размер | Decode |
|---|---:|---:|---|
| `DATEFORM/TIMEFORM` | preferred `0x06`, затем `0x07`, `0x04`, `0x08` | 5 байт | окно `DD MM YY HH MM`, decimal-first |
| `KODOPERA` | `0x05` | u8 | формат `02d` |
| `CODEDEF` / `К-во деф` | `0x0C` | u8 | `0..99` строкой |
| `NUMOBJ` body-only | `0x10`, `0x11`, `0x12`, `0x14` | `0x0B` | best reverse digit / PassportLUT fallback |
| `NUMOBJ` addr_len_truncated | `0x13`, `0x14`, `0x10`, `0x11` | `0x0B` | best reverse digit / PassportLUT fallback |
| `SMELTING` | `0x33`, `0x34`, `0x35`, `0x36` | `0x07` | best reverse digit / PassportLUT fallback |
| `TYPEVAR` body-only | `0x29`, `0x28`, `0x2A` | LE16 scan | `detect_typevar_code` |
| `TYPEVAR` addr_len_truncated | `0x2D`, `0x2C`, `0x29`, `0x28` | LE16 scan | `detect_typevar_code` |
| `PROTOCOL` primary | `0x21` | `0x07` | `reverse_digit_field`, digits only, strip right zeros |
| `PROTOCOL` fallback body-only | `0x21`, `0x22` | `0x07` | best reverse digit |
| `PROTOCOL` fallback prefix/truncated | `0x21`, `0x25` | `0x07` | best reverse digit |

Важная правка v34: `raw[0x0C]` — это `CODEDEF/К-во деф`, а **не** оператор. Оператор — `raw[0x05]`.

## 11. RESULTS2 / A-scan protocol v34

### 11.1 Принимаемые адреса и длины

| Диапазон | Смысл | Expected |
|---:|---|---:|
| `4000..4999` | A-scan RESULTS2 | `0x02B6` |
| `6000..6999` | extended A-scan RESULTS2 | `0x03A6` |

Минимум для попытки decode: `len(record) >= 0x40`. Если `len(record) < 0x60`, v34 помечает протокол как `неполный RAW; перечитать` и декодирует только безопасные link-поля.

### 11.2 Protocol → setting

```text
setting_no = protocol_setting_no_643(record)
setting_addr = 1000 + setting_no
```

Алгоритм:

| Случай | Offsets |
|---|---|
| Short 6000+n, `len(record)<0x60` | `0x0C`, `0x0E`, `0x10`, `0x0D`; первый `0 < LE16 <= 999` |
| Общий case | `0x0D`, `0x0C`, `0x1C`, `0x1D`; первый `0 < LE16 <= 999` |

Это live-коррекция v34. Старое статическое правило `record+0x1D` остаётся fallback, но не primary.

### 11.3 Protocol → graph

```text
if 4000 <= addr <= 4999: graph_addr = addr
if 6000 <= addr <= 6999: graph_addr = addr
```

v34 больше не делает искусственный `4000+n ↔ 6000+n` fallback для графика. График строится из самой считанной записи `graph_raw`; если отдельный graph record не найден, detail window берёт `protocol_raw`.

### 11.4 Поля A-scan protocol

| Поле GUI | Offset(s) | Размер | Decode |
|---|---:|---:|---|
| `DATEFORM/TIMEFORM` | preferred `0x06`, `0x07`, `0x04`, `0x08` | 5 | `best_datetime`; fallback `live_report_date_time` |
| `KODOPERA` | `0x05` | u8 | `02d` |
| `SPECIAL` | `0xF0` | u8 | `да`, если `record[0xF0] == 3` |
| `speed` для дефекта | `0x77` | LE16 | если не `1000..10000`, fallback `5900` |
| `TYPEVAR` | `0x29`, `0x2D`, `0x3A`, `0x3C` | LE16 scan | `detect_typevar_code`, с fallback по setting_no |
| `NUMOBJ` | `0x10`, `0x11`, `0x12`, `0x14` | `0x0B` | best reverse digit / PassportLUT fallback |
| `SMELTING` | `0x33`, `0x34`, `0x35`, `0x36` | `0x07` | best reverse digit; fallback `"0"` |
| `INDMAKER` | `0x3C` | LE16 | если !=0, строка |
| `MAKETIME` | `0x3E` | LE16 | если !=0, строка |
| `DEFEKT` | не raw[0x60] | логика defect table | `есть` только если CODEDEF реально map'ится |
| `CODEDEF` | `0x21` или `0x22` | `0x07` | reverse digits + original defect table |
| `SETTING_NO` | см. выше | LE16 | строка или пусто |
| `SETTING_ADDR` | `1000 + SETTING_NO` | — | строка или пусто |
| `GRAPH_ADDR` | `addr` | — | строка |

### 11.5 Defect logic v34

`decode_defect_643()` больше **не** использует `raw[0x60]` как флаг/координату дефекта. Комментарий в коде: `raw[0x5E]` — object word, `raw[0x60..]` — side/neck/rim area. Поэтому `raw[0x60]` не должен делать `DEFEKT='есть'`.

Подтверждённый путь:

```text
numeric_code = best_reverse_digit_field(record, ((0x21,0x07),(0x22,0x07)), max_len=8)
code_text = PassportLUT(record,0x21,0x07)
если numeric_code маппится через original defect table -> defect_present=True
иначе DEFEKT='нет'
```

### 12.2 Как выбирается offset графика

`find_ascan_graph_offset_643(record)` строит кандидаты:

| Кандидат | Условие/смысл |
|---:|---|
| `len(record) - 0xF4` | основной live-кандидат: graph = хвост записи |
| `0x1B8` | старый fallback `GRAPH_OFF` |
| `0x1C5` | fallback descriptor/raw-ish |
| `0x1E5` | fallback nominal zapis2 descriptor working |
| `len(record) - 0x100 - 0xF4` | fallback для extended layout |

Дубликаты offset удаляются только внутри списка кандидатов. Затем выбирается кандидат с максимальным score.

### 12.3 Score graph block

```text
samples = block[0:0xF3]
uniq = count(unique(samples))
vmin = min(samples)
vmax = max(samples)
near_baseline = count(0x40 <= sample <= 0xF0)
pad = count(sample in {0x00,0xFF})
score = uniq*3 + (vmax-vmin) + near_baseline - pad*4
```

### 12.4 Decode graph block

```text
off = best graph offset
block = record[off : off+0xF4]
samples = block[0:0xF3]
amplitudes = [sample - 0x8C for sample in samples]
min_sample = min(samples)
max_sample = max(samples)
line_mode = (record[0x128] & 1) == 0
special_geometry = record[0xF0] == 3
```

`block[0xF3]` копируется в `raw_block`, но не идёт в `samples`/отрисовку.

### 12.5 Отрисовка canvas

Canvas scaling в `ProtocolDetail.draw_graph()`:

```text
plot area = canvas minus margins ml=48, mr=24, mt=24, mb=38
baseline y = y_of_sample(0x8C)
amp = sample - 0x8C
y = mt + ph*0.5 - (amp/140.0) * (ph*0.5)
x sample i = ml + i/(0xF3-1)*pw
```

Зоны рисуются не по индексу sample, а по raw time coordinate:

```text
x_by_raw(raw_x) = ml + (raw_x / duration_t10) * pw
```

`duration_t10` берётся из `decode_ascan_zones_643()`, fallback `7920`.

### 13.2 Duration for zones

`duration_t10_from_record(record)` проверяет offsets:

| Offset | Смысл |
|---:|---|
| `0xA8` | live protocol #4002: duration shifted from setting `0x58` |
| `0x58` | setting/live-prefixed duration |
| `0x5A` | fallback/delay-adjacent |

Берётся первый `LE16` в диапазоне `1..2000`, затем:

```text
duration_t10 = round_half_up(sweep12_to_time_us(raw) * 10)
sweep12_to_time_us(raw) = raw * (0xF0 >> scale_shift) / 10
```

Default/fallback для x-coordinate: `duration_t10 = 7920`.

Y_mm = R_mm * cos(angle_rad)
X_mm = R_mm * sin(angle_rad)

Формат строки: `Y...мм,R...мм,Т...мкс`.

## 20. Главные расхождения с прежними заметками, чтобы не смешивать

| Тема | Раньше / статический вывод | v34 live-модель |
|---|---|---|
| Serial parity | комментарии reverse говорят `8E1` | live path использует `19200 8N1` |
| protocol→setting | primary `record+0x1D` | primary `0x0D`, fallback `0x0C/0x1C/0x1D`; short 6000 имеет свой порядок |
| graph link | `4000+n → 6000+n` | `graph_addr = addr`; график из самой записи |
| graph offset | fixed `0x1D5/0x1B8` | candidate scoring, основной live-кандидат `tail-0xF4` |
| report short `0x52..0x55` | можно было считать почти валидным | incomplete diagnostic, не decode |
| raw storage | только normalized | `raw_events` exact wire + `raw_records` normalized/latest |
| retry | были fallback/retry | v34 exact original: no retry |
| DTR/RTS | старые варианты трогали линии | v34 не трогает, driver-default |
| raw[0x60] defect | использовался как признак/координата | запрещено; только CODEDEF около `0x21` + defect table |

## 22. Что считать каноном при дальнейшем реверсе

1. Для живого обмена использовать v34 exact-55 model.
2. Все raw-файлы анализировать в двух слоях: `wire_raw` и normalized `record`.
3. График искать через candidate scoring, а не фиксировать один offset без проверки.
4. Любые отчёты длиной меньше declared длины не считать валидной дешифровкой.
5. Если offset из статического reverse конфликтует с live v34, подписывать базу offset: `zapis2 working`, `DLL body`, `record`, `wire`, `live shifted`.
